# 온라인 영화 예매 시스템 — 비기능적 요구사항 요약

# 1. 성능(Performance)

* 일반 API 는 빠른 응답 속도를 제공해야 한다.
* 좌석 조회 및 예매 API 는 실시간 수준의 응답 속도를 보장해야 한다.
* 예매 오픈 시 대규모 동시 접속 트래픽을 처리할 수 있어야 한다.
* 대량 요청 상황에서도 시스템 응답 지연이 최소화되어야 한다.
* 캐시 및 비동기 처리를 통해 조회 성능을 최적화해야 한다.

확인 필요:

* 실제 TPS 및 응답 시간 SLA 는 서비스 규모에 따라 달라질 수 있음.

Confidence: Medium (84%)

---

# 2. 확장성(Scalability)

* 서비스별 독립적인 Scale-Out 이 가능해야 한다.
* Booking, Search, Payment 와 같은 고부하 서비스는 개별 확장이 가능해야 한다.
* Stateless 기반 구조를 지원해야 한다.
* Kubernetes 기반 Auto Scaling 을 지원할 수 있어야 한다.
* 특정 기능의 부하 증가가 전체 시스템 성능 저하로 이어지지 않아야 한다.

---

# 3. 가용성(Availability)

* 시스템은 24시간 무중단 운영을 지원해야 한다.
* 일부 서비스 장애 시 전체 시스템 장애로 확산되지 않아야 한다.
* 장애 발생 시 자동 복구(Self-Healing)를 지원해야 한다.
* DB 및 메시지 브로커 이중화 구성이 필요하다.
* 장애 상황에서도 핵심 기능은 최대한 유지되어야 한다.

---

# 4. 데이터 정합성(Data Consistency)

* 동일 좌석의 중복 예매가 발생하지 않아야 한다.
* 결제 상태와 예매 상태 간 데이터 정합성을 유지해야 한다.
* 이벤트 중복 처리에 대한 Idempotency 를 보장해야 한다.
* 분산 환경에서 Eventually Consistency 를 지원해야 한다.
* 메시지 유실 및 중복 소비 대응 메커니즘이 필요하다.

---

# 5. 동시성 제어(Concurrency Control)

* 동일 좌석에 대한 동시 예매 요청을 안전하게 처리해야 한다.
* 좌석 Lock 및 Lock Timeout 기능을 제공해야 한다.
* 분산 환경에서 Distributed Lock 전략을 지원해야 한다.
* Race Condition 발생을 방지해야 한다.

---

# 6. 보안(Security)

* 모든 API 통신은 HTTPS 기반이어야 한다.
* JWT/OAuth2 기반 인증 체계를 제공해야 한다.
* 관리자 기능에 대한 권한 검증이 필요하다.
* 개인정보 및 민감 데이터는 암호화 저장해야 한다.
* 결제 정보 보호 정책을 준수해야 한다.

확인 필요:

* PCI-DSS 준수 범위는 실제 PG 연동 구조에 따라 달라질 수 있음.

Confidence: Medium (79%)

---

# 7. 장애 대응(Fault Tolerance)

* 서비스 간 Timeout 및 Retry 정책을 지원해야 한다.
* Circuit Breaker 를 통해 장애 전파를 방지해야 한다.
* Dead Letter Queue(DLQ)를 지원해야 한다.
* 메시지 브로커 장애 상황에서도 복구 가능해야 한다.
* 장애 발생 시 Fail Fast 전략을 적용할 수 있어야 한다.

---

# 8. 관측성(Observability)

* 중앙 집중식 로그 수집 체계를 제공해야 한다.
* 서비스별 메트릭 수집 및 모니터링이 가능해야 한다.
* Distributed Tracing 을 지원해야 한다.
* 실시간 모니터링 Dashboard 를 제공해야 한다.
* 장애 감지 및 알림 체계를 지원해야 한다.

---

# 9. 유지보수성(Maintainability)

* 서비스별 독립 배포가 가능해야 한다.
* 서비스 간 결합도를 최소화해야 한다.
* API 및 이벤트 계약(Contract)을 명확히 관리해야 한다.
* CI/CD 기반 자동 배포를 지원해야 한다.
* 장애 원인 분석 및 추적이 가능해야 한다.

---

# 10. 운영성(Operability)

* 운영자가 시스템 상태를 실시간으로 확인할 수 있어야 한다.
* 예매/결제/장애 현황을 조회할 수 있어야 한다.
* 로그 및 이벤트 기반 운영 분석이 가능해야 한다.
* 운영 중 서비스 무중단 배포가 가능해야 한다.

---

# 11. MSA 아키텍처 요구사항

* Database per Service 원칙을 준수해야 한다.
* 서비스 간 직접 DB 접근을 금지해야 한다.
* Event Driven Architecture 를 지원해야 한다.
* 서비스 간 Loose Coupling 구조를 유지해야 한다.
* API Gateway 기반 진입 구조를 제공해야 한다.

Confidence: High (97%)
